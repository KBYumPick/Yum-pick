# Business Rules - Unit 3: 장바구니&주문 (유진)

---

## 장바구니 규칙

### BR-CART-01: 단일 메뉴 최대 수량
- 장바구니 내 동일 메뉴의 수량은 **최대 99개**
- 수량 증가 시 99 초과 불가 - 버튼 비활성화 또는 무시
- 직접 입력 시 99 초과 값은 99로 클램핑

### BR-CART-02: 최소 수량
- 장바구니 내 메뉴 수량은 **최소 1개**
- 수량 감소 시 1 미만이 되면 해당 항목을 장바구니에서 제거
- (수량 0으로 설정 = 삭제와 동일)

### BR-CART-03: 동일 메뉴 중복 추가
- 이미 장바구니에 있는 메뉴를 다시 추가하면 수량 +1 증가
- 새 항목으로 추가하지 않음
- 수량 증가 후 99 초과 시 BR-CART-01 적용

### BR-CART-04: localStorage 동기화
- 장바구니 상태 변경(추가/수정/삭제/비우기) 시 즉시 localStorage에 저장
- 키: `yumpick_cart`
- 앱 초기화 시 localStorage에서 장바구니 복원
- localStorage 파싱 실패 시 빈 배열로 초기화 (에러 무시)

### BR-CART-05: 장바구니 비우기
- 주문 성공 후 자동으로 장바구니 비우기 (clearCart 호출)
- localStorage의 `yumpick_cart` 항목도 함께 삭제
- 주문 실패 시 장바구니 유지 (비우지 않음)

### BR-CART-06: 가격 스냅샷
- 메뉴를 장바구니에 담는 시점의 unitPrice를 저장
- 이후 관리자가 메뉴 가격을 변경해도 장바구니의 가격은 변경되지 않음
- 주문 생성 시 장바구니의 스냅샷 가격을 그대로 사용

## 주문 생성 규칙

### BR-ORDER-01: 빈 장바구니 주문 불가
- 장바구니가 비어있으면 주문 생성 불가
- 프론트엔드: "주문하기" 버튼 비활성화
- 백엔드: items 배열이 비어있으면 400 Bad Request 반환

### BR-ORDER-02: totalAmount 서버 검증
- 클라이언트가 전송한 totalAmount와 서버에서 계산한 금액이 일치해야 함
- 서버 계산: `items.reduce((sum, item) => sum + item.unitPrice * item.quantity, 0)`
- 불일치 시 400 Bad Request: "총 금액이 일치하지 않습니다."
- 허용 오차: 없음 (정확히 일치해야 함)

### BR-ORDER-03: 주문 항목 수량 검증
- 각 OrderItem의 quantity는 1 이상 99 이하
- 위반 시 400 Bad Request

### BR-ORDER-04: 세션 유효성
- 주문 생성 시 JWT의 sessionId와 요청 body의 sessionId가 일치해야 함
- 불일치 시 401 Unauthorized

### BR-ORDER-05: 주문 생성 후 처리
- 주문 성공 시: 주문 번호 표시 → 장바구니 비우기 → 5초 후 메뉴 화면 리다이렉트
- 주문 실패 시: 에러 메시지 표시, 장바구니 유지

## 주문 번호 생성 규칙

### BR-ORDNUM-01: 형식
- 형식: `YYYYMMDD-NNN` (날짜 8자리 + 하이픈 + 순번 3자리)
- 예: `20260331-001`, `20260331-042`

### BR-ORDNUM-02: 매장별·날짜별 순번
- 순번은 매장(storeId)별, 날짜별로 독립적으로 관리
- 매일 자정 이후 첫 주문은 001부터 시작
- 다른 매장의 주문 번호와 무관

### BR-ORDNUM-03: 순번 범위
- 순번은 001 ~ 999 (3자리 패딩)
- 하루 999건 초과 시: MVP 단계에서는 별도 처리 없음 (1000, 1001... 으로 자연 증가)

## 주문 조회 규칙

### BR-QUERY-01: 세션 기준 조회
- 고객앱의 주문 내역은 현재 sessionId 기준으로만 조회
- 다른 세션(이전 방문 고객)의 주문은 표시하지 않음
- 쿼리: `{ storeId, tableId, sessionId }`

### BR-QUERY-02: 정렬 순서
- 주문 내역은 createdAt 오름차순 정렬 (오래된 주문이 위)
- 동일 시각 주문은 orderNumber 오름차순

### BR-QUERY-03: 상태 필터 (선택)
- listOrders API는 status 파라미터로 필터링 가능
- 고객앱에서는 필터 없이 전체 조회 (현재 세션 기준)

## 총액 계산 규칙

### BR-TOTAL-01: 계산 방식
- totalAmount = 각 항목의 (unitPrice × quantity) 합산
- 소수점 없음 (원화 기준, 정수)
- 계산 시점: 장바구니 상태 변경 시마다 실시간 재계산

### BR-TOTAL-02: 표시 형식
- 천 단위 콤마 구분: `12,000원`
- 0원 표시: `0원` (빈 장바구니)

### BR-TOTAL-03: 서버-클라이언트 일관성
- 클라이언트 계산값을 서버에 전송
- 서버에서 재계산 후 검증 (BR-ORDER-02 참조)
- 저장되는 totalAmount는 서버 검증 통과 후의 값
